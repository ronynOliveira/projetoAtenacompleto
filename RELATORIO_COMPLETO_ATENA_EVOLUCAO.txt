1. VISAO GERAL DO PROJETO
================================================================================

Atena Evolucao e um sistema de IA cognitivo 100% local, projetado para rodar
em hardware modesto (Intel i5-1235U, 15.7GB RAM, sem GPU dedicada) usando
Ollama como motor de inferencia.

OBJETIVO PRINCIPAL:
  Criar um agente de IA que seja extensao da mente do usuario (Senhor Roberio),
  capaz de lembrar, aprender, criar e ajudar de forma proativa - sem depender
  de APIs externas pagas.

DIFERENCIAL:
  - 100% local (Ollama + Python + SQLite)
  - Custo ZERO de API
  - Privacidade total (nada sai da maquina)
  - Memoria de longo prazo com decay inteligente
  - Sistema de skills plugavel
  - UI web integrada
  - Seguranca em 7 camadas

3.1 MODELO DE IA LOCAL
----------------------

Modelo principal: atena-glm5 (3.8B parametros, baseado em phi4-mini)
  - Criado via Ollama com Modelfile customizado
  - System prompt em portugues com identidade "Atena"
  - Temperature: 0.85, top_p: 0.92, repeat_penalty: 1.15
  - Contexto: 8192 tokens (configuravel ate 131072)

Modelos auxiliares no Ollama:
  - phi4-mini (3.8B) - base para atena-glm5
  - gemma4:e2b (6.7GB) - tarefas que precisam de mais contexto
  - gemma4:e4b (8.9GB) - tarefas complexas
  - qwen3:8b (4.9GB) - alternativa leve
  - hermes3:8b (4.3GB) - modelo de chat
  - nomic-embed-text (0.3GB) - embeddings
  - deepseek-r1:8b (4.9GB) - raciocinio

3.2 SISTEMA DE MEMORIA DE LONGO PRAZO
-------------------------------------

Tecnica: Arquitetura 3 Camadas (inspirada em A-MEM, MemoryBank, LightMem)

CAMADA 1 - Memoria de Trabalho (Working Memory):
  - Buffer da sessao atual, em processo
  - Nao persiste entre sessoes
  - Implementacao: conversation_history em AtenaBridge

CAMADA 2 - Memoria Episodica (SQLite):
  - Notas atomicas estilo Zettelkasten / A-MEM
  - Cada nota: embedding + entidades + importancia + forca S
  - Decay: R = e^(-t/S) [MemoryBank / Ebbinghaus]
  - t = dias desde ultimo acesso (nao desde criacao)
  - S = forca, comeca em 1.0, aumenta a cada recall
  - Memorias abaixo de threshold (0.05) sao arquivadas (nao destruidas)

CAMADA 3 - Memoria Semantica (Fatos Consolidados):
  - Clusters de memorias episodicas relacionadas
  - 1 chamada ao LLM por cluster (nao por memoria individual)
  - Decai mais devagar (forca inicial mais alta)
  - Extracao de entidades via regex (nomes proprios + termos de dominio)
  - Clustering por overlap (union-find, O(n·e))

Tecnicas especificas:
  - Decay temporal: R = e^(-t/S) com S incrementado a cada recall
  - Spacing effect: memorias revisitadas resistem mais ao esquecimento
  - Token budget: retrieve_top_k() com limite de 400 tokens
  - Manutencao offline: 1x/dia (decay + arquivamento + consolidacao)

3.3 COMPORTAMENTO DO AGENTE
---------------------------

3.3.1 System Prompt Hierarquico (6 camadas):
  - Camada 0: Protocolo Fundacional (imutavel)
  - Camada 1: Identidade (imutavel)
  - Camada 2: Seguranca e Limites (imutavel)
  - Camada 3: Competencia e Estilo (mutavel)
  - Camada 4: Contexto de Sessao (mutavel)
  - Camada 5: Instrucao Imediata (mutavel)

3.3.2 Classificador de Tarefas (classify_task_v2):
  - 5 tipos: criativo, analitico, tecnico, factual, geral
  - Keywords expandidas por tipo
  - Temperatura adaptativa por tipo:
    * criativo: 0.9
    * analitico: 0.5
    * tecnico: 0.3
    * factual: 0.2
    * geral: 0.7

3.3.3 Constitutional AI:
  - Auto-critica apos cada resposta
  - Deteccao de incerteza, respostas curtas, ingles
  - Retry com prompt corrigido se issues encontradas

3.4 RAG (RETRIEVAL-AUGMENTED GENERATION)
-----------------------------------------

Tecnicas implementadas:
  - Chunking Semantico Adaptativo (256-512 tokens, overlap 15%)
  - Reranking via LLM (gemma4:e2b)
  - HyDE (Hypothetical Document Embeddings)
  - RAG Fusion (Multi-Query + Reciprocal Rank Fusion)
  - CRAG (Corrective RAG com avaliacao de relevancia)
  - Embeddings: nomic-embed-text via Ollama
  - Cache SQLite de embeddings

3.6 SEGURANCA
-------------

7 camadas de protecao:
  1. Input Sanitizacao (prompt injection, path traversal, command injection)
  2. Output Validacao (vazamento de dados, alucinacoes)
  3. Rate Limiting (token bucket, 30/min)
  4. Audit Log (SQLite com hash, timestamp, latencia)
  5. Model Integrity (checksum SHA-256 dos modelos)
  6. Secure Context (limpeza periodica, limite de tamanho)
  7. Anomaly Detection (7 detectores)

Medidas adicionais:
  - Content Security Policy (CSP)
  - Funcoes sanitizeHTML() e escapeHTML()
  - HTTPException sem str(e)
  - Validacao de max_length em Pydantic models

5. COMO COLOCAR O AGENTE HERMES NO NUCLEO DE UM LLM 4B+2B
================================================================================

5.1 PROBLEMA
------------

O modelo atena-glm5 (3.8B) precisa de:
  - ~2.3GB RAM (modelo) + ~1GB (Ollama overhead)
  - System prompt: ~2000 tokens
  - Memoria + RAG: variavel (ate 4000 tokens)

Em 3.8GB RAM (VPS), nao ha espaco suficiente para Ollama + modelo 4B +
Hermes Gateway + SO simultaneamente.

Em 15.7GB RAM (local), ha espaco com folga.

5.2 SOLUCOES POSSIVEIS
----------------------

SOLUCAO A: Modelo Menor (2B) na VPS
  - Usar qwen3:2b (1.2GB)
  - System prompt reduzido (500 tokens)
  - Memoria via RAG apenas
  - Vantagem: cabe em 3.8GB
  - Desvantagem: qualidade inferior

SOLUCAO B: Modelo 4B Local + Relay na VPS
  - Modelo 4B roda no Windows (15.7GB)
  - VPS serve apenas como relay/messaging
  - Vantagem: qualidade maxima
  - Desvantagem: depende do Windows estar ligado

SOLUCAO C: Quantizacao Extrema (Q2_K)
  - Quantizar modelo 4B para Q2_K (~1.5GB)
  - Perda de qualidade significativa
  - Vantagem: roda na VPS
  - Desvantagem: qualidade muito inferior

SOLUCAO D: Arquitetura Hibrida (RECOMENDADA)
  - VPS: modelo 2B (1.2GB) para tarefas simples
  - Local: modelo 4B (2.3GB) para tarefas complexas
  - Sincronizacao via EPR Bridge (WebSocket)
  - Vantagem: equilibrio entre qualidade e disponibilidade
  - Desvantagem: complexidade de sincronizacao

5.3 TECNICAS DE OTIMIZACAO DE CONTEXTO
--------------------------------------

1. System Prompt Dinamico: carregar apenas skills relevantes (200-500 tokens)
2. Memoria Seletiva: retrieve_top_k com token_budget=400
3. Compressao de Historico: resumos de conversas antigas
4. Lazy Loading de Skills: carregar sob demanda
5. Prompt Caching: cache de prompts frequentes no Ollama
